# Dynamic Programming

Dynamic Programming (DP) is often considered one of the most intimidating topics in computer science. But at its core, it is simply an optimization technique. 
DP is used to solve problems that can be broken down into **overlapping subproblems** which exhibit **optimal substructure**.

Instead of solving the same subproblem repeatedly (like naive recursion does), DP solves each subproblem once and stores the result.

## Real-world Analogy

Imagine I ask you: *"What is 1 + 1 + 1 + 1 + 1 + 1 + 1 + 1?"*
You count them up and say: *"8"*.

Then, I add one more `+ 1` to the end and ask: *"What about that?"*
You immediately say *"9"*.

How did you know it was 9 so fast? You didn't recount all the ones from the beginning. You remembered that the previous total was 8, and you just added 1. **That is Dynamic Programming.** 
You stored the answer to a previous subproblem to quickly solve a new, slightly larger problem.

<br />

## The Two Requirements for DP

To use Dynamic Programming, a problem must possess two properties:

1. **Overlapping Subproblems:** The problem can be broken down into smaller pieces, and these pieces are evaluated multiple times. (e.g., in Fibonacci, `fib(3)` is called multiple times).
2. **Optimal Substructure:** The optimal solution to the main problem can be constructed from the optimal solutions of its subproblems. (e.g., the shortest path from A to C via B is the shortest path from A to B + the shortest path from B to C).

<br />

## The Two Approaches

There are two ways to implement a DP solution: Top-Down and Bottom-Up.

### 1. Top-Down (Memoization)
You start from the main problem, break it down into subproblems, and solve them recursively. Whenever you solve a subproblem, you store its result in a cache (like a hash map or array). Before calculating a subproblem, you check if it's already in the cache.
- **Pros:** Intuitive if you already know how to write the recursive solution. Only computes necessary subproblems.
- **Cons:** Overhead of recursive function calls (call stack memory).

### 2. Bottom-Up (Tabulation)
You start from the smallest possible subproblems and build your way up to the main problem, usually using an iterative loop and an array to store results.
- **Pros:** No recursion overhead. Highly efficient.
- **Cons:** Requires figuring out the exact order in which to solve subproblems. Might compute subproblems that aren't strictly necessary.

<br />

## Example: Fibonacci Sequence

The Fibonacci sequence is the classic DP example. $F(n) = F(n-1) + F(n-2)$, with base cases $F(0) = 0$ and $F(1) = 1$.

### Naive Recursion (Terrible)
Time Complexity: $O(2^N)$

```python
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)
```

### Approach 1: Top-Down (Memoization)
Time Complexity: $O(N)$ | Space Complexity: $O(N)$

```python
def fib_memo(n, cache=None):
    if cache is None: cache = {}
    if n in cache: return cache[n]
    
    if n <= 1:
        return n
        
    # Store result before returning
    cache[n] = fib_memo(n - 1, cache) + fib_memo(n - 2, cache)
    return cache[n]
```

### Approach 2: Bottom-Up (Tabulation)
Time Complexity: $O(N)$ | Space Complexity: $O(N)$

```python
def fib_tab(n):
    if n <= 1: return n
    
    dp = [0] * (n + 1)
    dp[1] = 1
    
    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]
        
    return dp[n]
```

### Approach 3: Space-Optimized Bottom-Up
Time Complexity: $O(N)$ | Space Complexity: $O(1)$
Since we only ever need the *last two* values to calculate the next one, we don't need a whole array!

```python
def fib_opt(n):
    if n <= 1: return n
    
    prev2, prev1 = 0, 1
    
    for _ in range(2, n + 1):
        curr = prev1 + prev2
        prev2 = prev1
        prev1 = curr
        
    return prev1
```

## Problem Solving Framework

When approaching a DP problem, follow this 4-step framework:

1. **Define the Objective Function:** What does $DP[i]$ or $DP[i][j]$ represent? (e.g., $DP[i]$ = the max profit on day $i$).
2. **Identify Base Cases:** What are the simplest subproblems? (e.g., $DP[0] = 0$).
3. **Write the State Transition Equation:** How do you relate $DP[i]$ to $DP[i-1]$ or $DP[i-2]$?
4. **Determine the Order of Computation:** (Usually left-to-right, $0$ to $N$).

**Example: Climbing Stairs**
You are climbing a staircase. It takes `n` steps to reach the top. Each time you can either climb 1 or 2 steps. In how many distinct ways can you climb to the top?

1. **Objective:** $DP[i]$ = number of ways to reach step $i$.
2. **Base Cases:** $DP[0] = 1$ (1 way to stay at the ground), $DP[1] = 1$.
3. **Transition:** To get to step $i$, you either took a 1-step jump from step $i-1$, or a 2-step jump from step $i-2$. Therefore: $DP[i] = DP[i-1] + DP[i-2]$.
4. **Order:** Compute from $2$ to $n$.
*(Notice this is exactly the Fibonacci sequence!)*

## 1D vs 2D Dynamic Programming

Some problems require tracking multiple states at once. For example, if you are moving on a grid, your state is defined by your $(row, col)$ position.

**Example: Unique Paths**
A robot is located at the top-left corner of an $m \times n$ grid. It can only move down or right. How many unique paths are there to the bottom-right corner?

1. **Objective:** $DP[r][c]$ = number of paths to cell $(r, c)$.
2. **Base Cases:** The top row and left column only have 1 path (keep going right, or keep going down). $DP[0][c] = 1$, $DP[r][0] = 1$.
3. **Transition:** $DP[r][c] = DP[r-1][c] + DP[r][c-1]$.

```python
def uniquePaths(m, n):
    dp = [[1] * n for _ in range(m)]
    
    for r in range(1, m):
        for c in range(1, n):
            dp[r][c] = dp[r-1][c] + dp[r][c-1]
            
    return dp[m-1][n-1]
```

## Common Interview Questions

1. **Climbing Stairs:** The gateway to DP.
2. **Coin Change:** Minimum coins to make an amount. Introduces unbounded knapsack concept.
3. **Longest Increasing Subsequence:** Classic $O(N^2)$ DP problem.
4. **House Robber:** Maximizing sums without taking adjacent elements.
5. **Longest Common Subsequence:** A fundamental 2D DP string problem.
